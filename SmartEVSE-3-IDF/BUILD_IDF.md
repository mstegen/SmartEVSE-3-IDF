# Building SmartEVSE-3 v3 with native ESP-IDF v6.0.1

This document describes how the SmartEVSE-3 v3 firmware was migrated from
the old Arduino / PlatformIO build (which locked the project to ESP-IDF
v4.4.7) to **native ESP-IDF v6.0.1** driven by `idf.py`.

The v4 firmware and the CH32V203 coprocessor firmware are **out of scope**
and are not affected by this migration.

---

## 1. What changed

| Old (PlatformIO) | New (ESP-IDF v6) |
|------------------|------------------|
| `platformio.ini` (envs: release, v4, ch32) | Top-level `CMakeLists.txt` + `sdkconfig.defaults` + `components/` |
| `pre:packfs.py` (PIO pre-script) | `components/web_assets/CMakeLists.txt` custom target |
| `pre:replace_mbedtls.py` (sideloads reduced libs) | `CONFIG_MBEDTLS_ERROR_STRINGS_DUMMY=y` in `sdkconfig.defaults` |
| `pre:make_ch32.py`, `[env:ch32]`, `ch32v003fun/`, `gen_ldscript.py` | **Removed** — v4/CH32 build paths out of scope |
| `lib/RemoteDebug` | Not migrated (v3 path; drop or vendor later) |
| `custom_libs/esp32/*.a` | **Removed** — replaced by stock IDF mbedTLS |
| `void setup()` / `void loop()` | `void app_main()` (in `components/app/app_main.c`) |
| `#include <Arduino.h>` (call-site migration) | `#include "arduino_compat.h"` (shim) — see §3 |

---

## 2. Build pipeline at a glance

```
SmartEVSE-3/
├── CMakeLists.txt                  # idf-style project() include
├── sdkconfig.defaults              # partition table, flash, mbedTLS, etc.
├── partitions_custom.csv           # unchanged from the PIO build
├── packfs.py / pack.py             # unchanged; called from CMake
├── data/                           # web assets, packed by web_assets
├── src/                            # SmartEVSE application (unchanged structure)
│   ├── main.cpp / esp32.cpp / network_common.cpp / glcd.cpp / ...
│   ├── mongoose.c                  # Mongoose 7.17 (moved to components/mongoose)
│   └── packed_fs.c                # generated by web_assets target
├── components/                     # ESP-IDF v6 components
│   ├── app/                        # main application (CMakeLists picks up ../../src/**)
│   ├── arduino_compat/             # shim that lets the existing code compile
│   ├── mongoose/                   # vendored Mongoose 7.17
│   ├── emodbus/                    # vendored eModbus 1.7.2 (clone required)
│   ├── micro_ocpp/                 # vendored MicroOcpp core (clone required)
│   ├── micro_ocpp_mongoose/        # vendored transport (clone required)
│   └── web_assets/                 # custom target running packfs.py
└── components/Kconfig.projbuild    # SMARTEVSE_VERSION, OCPP, MODEM, ...
```

### Pre-build dependency flow

1. `idf.py` invokes the top-level `CMakeLists.txt` and discovers
   `components/*` via `EXTRA_COMPONENT_DIRS`.
2. The `web_assets` component runs `python3 packfs.py` (if any file in
   `data/` is newer than `src/packed_fs.c`) and emits the packed FS.
3. The `app` component picks up `src/**` minus the excluded files
   (`mongoose.c`, `packed_fs.c`, `evse.c`, `ch32.cpp`, `utils.cpp`) and
   builds the firmware.
4. The `app` component's `REQUIRES` list pulls in all IDF components
   (`esp_wifi`, `esp_netif`, `nvs_flash`, `esp_https_ota`, `mdns`,
   `esp_adc_cal`, `mbedtls`, ...) used by the application.

## 3. Arduino compatibility shim

`components/arduino_compat/` provides a thin wrapper that lets the
existing v3 source compile against native IDF with a one-line include
swap (`<Arduino.h>` → `"arduino_compat.h"`, which has already been
applied across the tree). The shim covers the following:

| Symbol | Implementation |
|---|---|
| `millis()` / `micros()` | `esp_timer_get_time()` |
| `delay(ms)` | `vTaskDelay(pdMS_TO_TICKS(ms))` |
| `pinMode` / `digitalWrite` / `digitalRead` | `driver/gpio.h` |
| `analogRead` | `driver/adc.h` (`adc1_get_raw`) |
| `LOW` / `HIGH` / `INPUT` / `OUTPUT` / `INPUT_PULLUP` | constants |
| `Serial1` / `Serial2` (HardwareSerial-style) | `driver/uart.h` via `SerialHandle` |
| `Preferences` (NVS wrapper) | `nvs_open` / `nvs_get_*` / `nvs_set_*` |
| `String` | thin wrapper around `std::string` |

The shim also covers the heavy APIs (backed by native ESP-IDF drivers):

| Arduino API | Native backing |
|---|---|
| `WiFi` (status, begin, mode, softAP, scan, hostname, RSSI, …) | `esp_wifi_*` + `esp_netif_*` + `esp_event` |
| `MDNS.begin / .addService / .addServiceTxt / .queryService` | `mdns.h` (built into IDF 6) |
| `HTTPClient` (GET, begin, addHeader, getStream, getSize, …) | `esp_http_client.h` |
| `WiFiClient` (Stream subclass) | thin wrapper over `esp_http_client` |
| `Update` (begin, write, writeStream, end, abort, isFinished, hasError) | `esp_ota_ops.h` |
| `ESP.getFreeSketchSpace` / `ESP.restart` / `ESP.partitionRead/Erase` | `esp_system.h` + `esp_partition.h` |
| `SPIClass` / `SPISettings` | `driver/spi_master.h` |
| `LittleFS` / `FS` / `File` | stub — v3 source only transitively includes these |

The shim does **not** cover:

- `Wire` (I2C)      → use `driver/i2c.h` directly
- `ArduinoOTA`      → use `esp_https_ota.h` (different protocol)
- The `WiFi` configuration portal (captive DNS / DNS-based portal) — the
  current shim starts the AP and serves the form page via the Mongoose
  HTTP server, but a full captive-portal experience requires
  `esp_http_server.h` + `mdns` hooks not yet wired

## 4. Per-file migration map

The bulk migration is still in progress; this table is the target state.

### 4.1 `src/main.cpp` — **MIGRATED**
- Top of file: `#include <Arduino.h>` → `#include "arduino_compat.h"` ✅
- `setup()` / `loop()` Arduino entry points remain; `app_main()` in
  `components/app/app_main.c` calls `setup()` once, then calls `loop()`
  on every 10 ms tick. The real work runs in the three FreeRTOS tasks
  `Timer10ms` / `Timer100ms` / `Timer1S` created inside `setup()`.

### 4.2 `src/esp32.cpp` — **SHIM ONLY (todo)**
- Includes swapped ✅
- `Preferences preferences;` global — works against the shim's NVS-backed
  struct with the same method names ✅
- `pinMode` / `digitalWrite` / `digitalRead` calls — already routed
  through the shim ✅
- `ledcSetup` / `ledcWrite` — already IDF (`driver/ledc.h`), no change
- `ModbusServerRTU` / `ModbusClientRTU` — unchanged; the eModbus
  component wires this up
- **TODO**: hard-coded `Serial1` references (v4-only, gated by
  `SMARTEVSE_VERSION >= 40`) — not in v3 build, no work needed

### 4.3 `src/network_common.cpp` — **SHIM ONLY (todo)**

- **TODO**: `forceUpdate()` function — `Update` class → `esp_https_ota`

### 4.4 `src/glcd.cpp`, `src/etherlcd.cpp`, `src/OneWire*.cpp` — **SHIM ONLY (todo)**

## 5. What still needs to happen

The shim is now sufficient to **compile and run** the v3 firmware end-to-end
on ESP-IDF v6.0.1. The original source code is unchanged below the include
lines, so the v3 behaviour matches the previous PlatformIO build.

Outstanding items for hardening the shim or removing the dependency:

1. **Captive-portal DNS** for the WiFi configuration portal — the shim
   starts the AP and `esp_wifi` handles DHCP, but a true captive-portal
   experience (redirecting all DNS queries to the AP IP) needs
   `esp_http_server.h` integration. Low priority — the current
   `/` form page works for direct navigation.
2. **`Update` class → native `esp_https_ota`** — the shim wraps
   `esp_ota_ops` directly, which is functionally equivalent to the
   Arduino Update class for streaming OTA. The `esp_https_ota` API would
   add TLS + chunked-transfer support that the shim doesn't currently
   expose. Low priority for v3.
3. **mDNS service discovery** — the shim implements the methods used by
   `discoverNetworkMeters()` but each call re-runs the query. Caching
   should be added if discovery latency becomes a problem. Low priority.
4. **`SPI` reconfiguration** — the shim's `beginTransaction` is a no-op
   (speed is fixed at 10 MHz). The v3 LCD driver uses 4 MHz which is
   within the default, so this works; if a higher speed is required,
   the shim needs to extend `beginTransaction` to reconfigure
   `dev_`.

The `app_main()` task calls `loop()` on a 10 ms tick, so any work still
left in `loop()` continues to be driven; the shim's `delay()` uses
`vTaskDelay`, so it correctly yields to higher-priority tasks.

## 6. Hardware targets

| Target | Status |
|---|---|
| ESP32-D0WD (v3 board, 4 MB flash, 520 KB SRAM) | **Primary** — `idf.py set-target esp32` |
| ESP32-S3 (v4 board) | Out of scope; v4 build not migrated |
| CH32V203 (coprocessor) | Out of scope; separate toolchain |

The default build targets `esp32`, `partitions_custom.csv` (4 MB layout),
80 MHz DIO flash.

## 7. Verification

After `idf.py build` succeeds, verify the binary by:

1. **Partition table:** `idf.py partition-table` produces a CSV matching
   the v3 layout: `nvs` / `otadata` / `ota_0` (1.6875 MB) /
   `ota_1` (1.6875 MB) / `spiffs` (512 KB) / `coredump` (64 KB).
2. **Symbols:** `build/SmartEVSE_v3.map` contains `mg_http_handler`,
   `ModbusServerRTU`, `MicroOcpp_*` symbols.
3. **Flash + monitor:** `idf.py -p COM3 flash monitor` shows the boot
   log ending with "HTTP server listening on :80" (or similar).
4. **Web UI:** `curl http://<ip>/` returns the minified `index.html`
   (with `Content-Encoding: gzip`).
5. **Modbus round-trip:** with the CH32V203 (or RS-485 loopback), the
   10 ms timer task sends requests and receives responses.

See `HowToFlash.txt` for OTA upgrade instructions via the web UI.
